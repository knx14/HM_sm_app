T1【バグ修正】時系列グラフの作業ログ縦線が表示されない timeseries vertical line

原因の特定と修正方針

Phase 5 の実装では fl_chart の ExtraLinesData / VerticalLine を想定したコードを書きましたが、 実際は graphic ライブラリを使って実装されました。 graphic では縦線の描画方法が異なるため、凡例は表示されても実際のラインが描画されない状態になっています。

修正前に必ず graphic のバージョンと現在の実装コードを確認してください。 graphic ライブラリのバージョン（v1 / v2）によって Annotation / Mark の API が大きく異なります。 Cursor に渡す前に grep "graphic" pubspec.yaml でバージョンを確認し、その結果をセッション冒頭に伝えてください。

Chart(
  data: chartData,          // 既存の List<Map> を維持
  variables: {
    'date': Variable(accessor: (Map row) => row['date'] as String),
    'avg':  Variable(accessor: (Map row) => row['avg']  as num),

// 作業ログ縦線の TagAnnotation コンテンツ
Widget _buildWorkLogAnnotation(WorkLogMark wl) {
  final color = wl.workType == 'fertilization'
      ? Colors.orange
      : wl.workType == 'tillage'
          ? const Color(0xFF7A4525)
          : Colors.purple;

  return Column(mainAxisSize: MainAxisSize.min, children: [
    // ラベル（縦書き or 短縮）
    Container(
      padding: const EdgeInsets.symmetric(horizontal: 3, vertical: 1),
      color: color.withValues(alpha: 0.15),
      child: Text(
        wl.workType == 'fertilization' ? '施肥'
            : wl.workType == 'tillage' ? '耕うん' : '作業',
        style: TextStyle(fontSize: 9, color: color, fontWeight: FontWeight.bold),
      ),
    ),
    // 縦線本体（Container で代用）
    Container(
      width: 1.5,
      height: 120,  // グラフ高さに応じて調整
      color: color.withValues(alpha: 0.7),
    ),
  ]);
}
graphic v1 と v2 の API 差分について。
  • v1: Annotation ではなく MarkAnnotation / RegionAnnotation を使う場合があります
  • v2: TagAnnotation が推奨。values に変数マップを渡します
  • いずれのバージョンでも、X 軸の変数名と値の型を Chart の variables 定義と一致させることが最も重要です

WorkLogNotifier に編集モード対応を追加

lib/features/work_logs/presentation/work_log_notifier.dart（編集） 
WorkLogNotifier に forEdit コンストラクタと save のPATCH分岐を追加  final int? _editWorkLogId;  // null = 新規, non-null = 編集

  // 編集用コンストラクタ（既存値で各フィールドを初期化）
  factory WorkLogNotifier.forEdit({
    required int          farmId,
    required int          workLogId,
    required WorkLogEntry initial,
    required WorkLogRepository repo,
  }) {
    final n = WorkLogNotifier(farmId: farmId, repo: repo, editWorkLogId: workLogId);
    // 既存値をフォームに反映
    n.workType    = WorkType.fromValue(initial.workType);
    n.workDate    = initial.workDate;
    n.title       = initial.title    ?? '';
    n.detail      = initial.detail   ?? '';
    n.amountText  = initial.amountValue?.toString() ?? '';
    n.amountUnit  = initial.amountUnit ?? 'kg';
    n.step        = 1;  // 編集時は Step1（種別選択）をスキップして Step2 から開始
    return n;
  }

  // save() に PATCH 分岐を追加
  Future<void> save() async {
    if (workType == null) return;
    isSaving = true; saveError = null; notifyListeners();

    final entry = WorkLogEntry(
      workType:    workType!.value, workDate: workDate,
      title:       title.isNotEmpty    ? title    : null,
      detail:      detail.isNotEmpty   ? detail   : null,
      amountValue: double.tryParse(amountText),
      amountUnit:  amountUnit.isNotEmpty ? amountUnit : null,
    );

    try {
      if (_editWorkLogId != null) {
        // 編集モード: PATCH
        await _repo.update(_editWorkLogId!, entry);
        saveQueued = false;
      } else {
        // 新規モード: POST（既存ロジック）
        final result = await _repo.create(_farmId, entry);
        saveQueued = result.queued;
      }
    } catch (e) {
      saveError = e.toString();
    } finally {
      isSaving = false;
      notifyListeners();
    }
  }

T3【機能追加】グラフ点タップで測定値詳細を表示 chart tap detail

設計判断

「時系列タブとタイムラインタブのどちらで詳細を見せるか」について判断します。

推奨: 時系列タブで対応します。 理由：時系列タブはグラフの「点」がある場所であり、タップした点が「その日の測定」であることが直感的に伝わります。 タイムラインは既にカードとして日付ごとの情報を表示しており、 そこに「タップで詳細」を追加すると情報の重複が生じます。 時系列タブの点タップで BottomSheet を開き、その日の全パラメータ（CEC・CaO・K₂O・MgO と、 将来的には全17種）を表示する設計が最もシンプルで使いやすいです。

Future<void> fetchUserName() async {
  try {
    final attrs = await Amplify.Auth.fetchUserAttributes();
    // 'name' 属性を優先、なければ 'preferred_username'、なければ email の @ 前
    final name = attrs
        .firstWhere((a) => a.userAttributeKey.key == 'name',
            orElse: () => attrs.firstWhere(
                (a) => a.userAttributeKey.key == 'preferred_username',
                orElse: () => attrs.firstWhere(
                    (a) => a.userAttributeKey.key == 'email',
                    orElse: () => throw 'no name attr')));
    final val = name.value;
    // email の場合は @ より前を名前として使う
    _userName = val.contains('@') ? val.split('@').first : val;
    notifyListeners();
  } catch (_) {
    _userName = _user?.userId.substring(0, 8) ?? 'ユーザー';
    notifyListeners();
  }
}

MeasurementSessionScreen の圃場初期化に最近傍選択を追加

測定画面が開いたとき（initState または既存の圃場候補ロード処理内）に、 GPS 現在地と各圃場の重心（または boundary_polygon の中心）との距離を計算し、 最も近い圃場を自動的に選択済みにします。 

最近傍圃場の自動選択ロジック（既存のfarmCandidates読み込み後に追加）import 'package:geolocator/geolocator.dart';

/// 圃場候補リストから現在地に最も近い圃場を返す。
/// 圃場の位置は boundary_polygon の重心（centroid）または
/// 既存の farms API レスポンスに含まれる緯度経度を使う。
Future<void> _autoSelectNearestFarm() async {
  if (farmCandidates.isEmpty) return;

  final pos = await Geolocator.getCurrentPosition(
    locationSettings: const LocationSettings(
      accuracy: LocationAccuracy.high,
      timeLimit: Duration(seconds: 5),  // タイムアウト5秒
    ),
  ).catchError((_) => null);   // GPS エラーは無視して続行
  if (pos == null) return;

  // 各圃場と現在地の距離を計算
  dynamic nearest;
  double minDist = double.infinity;

  for (final farm in farmCandidates) {
    // 圃場の代表座標を取得する方法（実装に合わせて変更）
    // パターン A: farm.latitude / farm.longitude がある場合
    final farmLat = (farm.latitude  as double?);
    final farmLng = (farm.longitude as double?);
    if (farmLat == null || farmLng == null) continue;

    // パターン B: boundary_polygon から重心を計算する場合は
    // _computeCentroid(farm.boundaryPolygon) を使う（既存ロジック流用）

    final dist = Geolocator.distanceBetween(
      pos.latitude, pos.longitude, farmLat, farmLng);

    if (dist < minDist) {
      minDist = dist;
      nearest = farm;
    }
  }

  if (nearest != null) {
    setState(() {
      // 既存の「圃場選択」チップの選択状態を更新（実際の変数名に合わせること）
      _selectedFarmId   = nearest.id as int;
      _selectedFarmName = nearest.farmName as String;
    });
  }
}
