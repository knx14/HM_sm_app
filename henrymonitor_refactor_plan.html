0本書の目的とスコープ scope

本書は、HenryMonitor モバイルアプリ（Flutter / Android）のUI大幅刷新および機能追加を、 Cursor を用いて段階的に実装させるための作業指示書である。 現状の3タブ構成（ホーム / 測定 / 圃場管理）から、 新画面遷移図 v0 に従う 5系統メニュー構成（測定 / 圃場 / 同期 / 設定 / ヘルプ） へ刷新する。

方針サマリー

  • UI刷新: ホーム画面を主要4ボタン構成に変更。測定画面を地図中心1画面化。圃場詳細を 3 タブ（マップ / 時系列 / タイムライン）化。
  • 機能追加: 作業ログ記録、タイムライン表示、未同期データ管理画面の独立化、ヘルプ画面の新設。
  • 時系列方針の変更: 過去議論では 固定地点 の時系列追跡を計画していたが、本改修では固定地点機能は導入しない。時系列タブで追う数値は 「その圃場における各測定点の平均値」 とする。
  • システム改修: 作業ログ用テーブル work_logs 新設。Laravel に作業ログCRUD APIおよび時系列/タイムライン取得APIを追加。
  • 非スコープ: Lambda推論パイプライン本体は無変更（Ca/K/Mg対応は別タスク）。iOS対応は対象外。
固定地点導入の見送り理由（記録）。 固定地点の同一性判定は GPS 精度（3m スレッショルド）に大きく依存し、UX 上の混乱が発生しやすい。 一方、圃場平均値の推移であっても、施肥・耕うん作業との相関分析という当初目的（ケース3：融合型）は十分に達成可能と判断した。 将来、固定地点機能を再導入する場合は、本書で構築する work_logs と analysis_results のスキーマを拡張する形で対応する。

2データベース改修 schema

2.1 新規テーブル work_logs

作業ログ（施肥・耕うん等）を保存するテーブルを新設する。圃場（farms）に紐づく。

CREATE TABLE work_logs (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  farm_id         BIGINT UNSIGNED NOT NULL,
  work_type       VARCHAR(32) NOT NULL,
                  -- 'fertilization' | 'tillage' | 'pesticide' | 'harvest' | 'other'
  work_date       DATE NOT NULL,
  title           VARCHAR(128) NULL,
                  -- 例: 'NK化成 20kg', '深さ15cm トラクター'
  detail          TEXT NULL,
                  -- 自由記述
  amount_value    DECIMAL(10,2) NULL,
                  -- 例: 20.0
  amount_unit     VARCHAR(16) NULL,
                  -- 'kg', 'L', 'cm' など
  scope           VARCHAR(16) NOT NULL DEFAULT 'whole',
                  -- 'whole' | 'partial'（将来用）
  created_at      TIMESTAMP NULL,
  updated_at      TIMESTAMP NULL,
  INDEX idx_work_logs_farm_date (farm_id, work_date),
  CONSTRAINT fk_work_logs_farm
    FOREIGN KEY (farm_id) REFERENCES farms(id) ON DELETE CASCADE
);
命名・enum 方針。 work_type は文字列カラムとし、Laravel 側で App\Enums\WorkType を定義して制御する。 DB レベル ENUM は将来の値追加で migration が面倒になるため避ける。

2.2 既存テーブルの変更点

基本的に既存スキーマは無変更。ただし以下を明記する。

  • analysis_results に カラム追加は行わない（固定地点フラグなどは導入しない）。
  • result_values の (analysis_result_id, parameter_name) UNIQUE 制約は既に存在する前提で進める。
  • uploads.status の遷移仕様（uploaded → processing → completed / exec_error）は変更しない。

2.3 migration 作成手順

  1. php artisan make:migration create_work_logs_table を実行し、上記 DDL を反映。
  2. App\Models\WorkLog を作成。$fillable と casts（work_date → date, amount_value → decimal）を設定。
  3. App\Models\Farm に workLogs(): HasMany を追加。
  4. 本番投入前にステージング DB で php artisan migrate --pretend で SQL を確認。

  "amount_unit": "kg",

GET /api/farms/{farmId}/results/timeseries?parameter=CEC

時系列タブ向け。指定パラメータについて、各測定日（uploads.measurement_date ベース）の平均/最小/最大を返す。

{
  "parameter": "CEC",
  "unit": "me/100g",
  "points": [
    { "date": "2026-02-05", "avg": 21.1, "min": 19.8, "max": 22.6, "count": 5 },
    { "date": "2026-03-10", "avg": 21.2, "min": 20.0, "max": 22.5, "count": 4 },
    { "date": "2026-03-28", "avg": 24.3, "min": 22.4, "max": 25.6, "count": 6 }
  ]
}
集計仕様（厳密）。 uploads.status='completed' のみを対象とする。同日に複数 upload がある場合は analysis_results 単位ですべての測定点を統合して avg/min/max を出す。 count は upload数ではなく測定点数（analysis_results 行数） を返す。

GET /api/farms/{farmId}/timeline

タイムラインタブ向け。測定結果と作業ログを時系列降順（新しい順）でマージ。

{
  "items": [
    {
      "type": "measurement",
      "date": "2026-03-28",
      "summary": {
        "count_points": 6,
        "values": {
          "CEC": { "avg": 24.3, "min": 22.4, "max": 25.6, "unit": "me/100g" }
        },
        "effect_delta_cec": 3.1
      }
    },
    {
      "type": "work_log",
      "date": "2026-03-16",
      "work_type": "fertilization",
      "title": "NK化成 20kg",
      "detail": "圃場全体"
    }
  ]
}
「効果（effect_delta_cec）」の算出ルール。 各測定結果アイテムについて、当該測定の直前の同パラメータ測定値との avg 差分を返す。 直前測定がない場合は null。 UI 表示時はプラス値を緑、マイナス値を赤で表示。

4.2 画面別実装指針

screen / homeホーム画面

  • 上部に Cognito ユーザーの名前を表示（タップで SettingsScreen へ）。
  • 「測定を始める」「圃場」を縦に並んだ大ボタンで配置。背景色は赤系（測定）と緑系（圃場）で視覚的に区別。
  • 下段に「同期」「設定」の 2 分割ボタン。同期ボタンには 未同期件数バッジと赤い注目アイコンを重ねる。0件時はバッジ非表示。
  • 最下部に細いバーで「ヘルプ」へのリンク。
  • 未同期件数は PendingUploadsService（既存）から StreamProvider 経由で監視。

screen / measurement測定画面（地図中心1画面）

  • 画面上部に3つのステータスチップ：圃場選択 / 接続 / BG。すべて完了で下部の「測定開始」ボタンが有効化。
  • 圃場選択は 現在地から候補をプルダウンで表示（GPS から最寄り N 件）。
  • 接続・BG は内部状態（USBシリアル接続結果・BG測定完了フラグ）と連動して色で表示。
  • 地図は航空写真ベース、選択中圃場の boundary を赤枠で描画。圃場外で測定開始しようとした場合は警告ダイアログを出す。
  • 測定済みピンは番号付き丸（赤 = 測定中/エラー、黄 = 測定完了、緑 = 推定完了）。地図上ではタップしても何もしない（誤操作防止）。
  • 右上「リスト」ボタンで MeasurementListScreen を開く。右下「現在地」ボタンで GPS にセンタリング。
ピン状態遷移の仕様。 local_save → uploaded(measure済) → completed(推定済) の3段階を、ローカルDB(SQLite/JSON)の状態と Laravel API /api/farms/{id}/results/map の照合で判定する。 タイムアウト時は赤（エラー）に落とす。

screen / measurement-list測定リスト画面（サブ）

  • 画面タイトル「測定リスト」と圃場名。件数表示。「未確定のみ」フィルタチップ。
  • 各行に番号付き丸（地図ピンと同色）、CEC/CaO/MgO/K₂O の値、ステータスラベル（測定中 / 測定完了 / 推定完了 / エラー）。
  • 未実装パラメータ（Ca/K/Mg）は当面 - 表示（プレースホルダー）。
  • 下部に「選択を削除」「位置を修正」ボタン。チェックボックスで複数選択。
  • 「位置を修正」では再度マップ上のタップで補正させ、ローカル状態のみ更新（既にアップロード済の場合は補正不可、UIで disable）。

screen / farm-list圃場一覧

  • カードレイアウト（3行構成）：圃場名・作物・面積 / 最終測定日・点数 / CEC・CaO・MgO・K₂O 平均値。
  • サムネイル左カラムに boundary 領域の航空写真スナップ（Static Map API、もしくはローカルキャッシュ）。
  • 上部に検索バー（圃場名・作物名 で前方一致フィルタ、クライアントサイド）。
  • 下部に「+ 新規圃場を登録」ボタン → FarmRegisterScreen。
  • カード長押しで編集/削除メニュー。

screen / farm-detail圃場詳細（3タブ）

共通ヘッダ：圃場名 / 作物・面積。下部に「+作業記録」ボタン（時系列・タイムラインタブのみ表示）。

① マップタブ

  • パラメータ切替タブ（CEC / Ca / K / Mg）。未実装は disable（グレーアウト）。
  • 測定日のプルダウンセレクタ（既存の /api/farms/{id}/results/dates を利用）。
  • 選択日の測定点を地図上に円で表示（値は数値ラベル、色は値域でグラデーション）。
  • 下部にカラースケール凡例とその日の平均値。

② 時系列タブ 新規

  • パラメータ切替タブ（CEC / Ca / K / Mg）。
  • 大きな数値で「圃場平均」を表示（最新測定日の値）。
  • 下に折れ線グラフ：横軸＝測定日、縦軸＝平均値。min/max を半透明バンドで表示。
  • 作業ログのタイミングを縦点線で重畳表示（凡例：「耕うん」「施肥」など色分け）。
  • 固定地点機能は搭載しない。 単純に各測定日の圃場全体平均を折れ線で結ぶ。
  • データソース：GET /api/farms/{id}/results/timeseries + GET /api/v1/farms/{id}/work-logs。
「圃場平均」の定義。 ある測定日の圃場平均値 = その日に completed 状態となった全 analysis_results の指定パラメータ値の単純平均。 測定点数が変動しても気にせず平均を取る。これがケース3（融合型）の主要可視化となる。

③ タイムラインタブ 新規

  • 「作業と測定の記録」ヘッダ。
  • カードを時系列降順（新しい順）に並べる。

  • 作業ログカード：日付・作業種別・タイトル・詳細・量と単位。
  • カラーチップで作業種別を色分け（施肥=オレンジ、耕うん=茶、農薬=紫、収穫=黄など）。
  • 下部「+作業記録」で WorkLogEditScreen を showModalBottomSheet で開く。

screen / work-log-edit作業記録入力

  • 3タップ以内で完了するテンプレ選択フローを目標とする。

  • Step2: 日付 + 量 + 単位 + タイトル（プリセット候補）の入力。
  • Step3: 確認 → 保存（POST /api/v1/farms/{id}/work-logs）。

  • 「全選択」トグル（ツールバー）。
  • カード：圃場名・点数・日時・状態ラベル（準備完了 / データ欠損 など）。エラーは赤背景で強調。
  • 下部に「選択を削除」「同期」の 2 ボタン。
  • 同期中は進捗バーとリトライボタン。タイムアウト時は自動リトライ（最大3回）。

screen / settings設定画面

  • セクション①「アカウント」：ユーザー名・メール・ログアウト。
  • セクション②「通信」：測定機器名・シリアル番号・自動同期条件（Wi-Fi のみ / 常時 / 手動）。優先度低
  • セクション③「システム」：測定条件の設定（既存の詳細設定画面を移植）・バージョン・利用規約。利用規約はダミーボタンでOK。優先度低
  • 従来 BgScreen や測定パラメータ詳細設定の機能は 「測定条件の設定」サブ画面に集約する。

screen / helpヘルプ画面

  • 4項目構成：使い方ガイド / FAQ / 問い合わせフォーム / アプリ情報。
  • 使い方ガイドは 5 ステップ（圃場登録 → 機器接続 → BG → 測定 → 同期）を 1 画面で見せる。
  • 問い合わせフォームはアプリ内で完結。送信時に端末情報・アプリバージョンを自動付与（フォーム上は非表示）。送信先 API は別途決定（暫定: POST /api/v1/inquiries）。
  • FAQ・アプリ情報・利用規約は当面ダミーで OK。優先度低

5.3 ローカル DB（オフライン対応）

作業ログのオフライン入力対応のため、sqflite 導入を検討する。導入を躊躇する場合は、既存の pending_uploads.json と同様の JSON キューファイル方式でも可。本改修では暫定として JSON キュー方式で進め、運用負荷を見ながら sqflite 化を判断する。

// pending_work_logs.json (例)
[
  {
    "local_id": "uuid-xxxx",
    "farm_id": 12,
    "payload": { /* POST body */ },
    "created_at": "2026-04-01T10:00:00+09:00",
    "synced": false
  }
]

8リスクと懸念事項 risks

リスク影響度緩和策
既存 BottomNav 廃止により、現行ユーザーが導線を失う 中 ホーム画面の主要4ボタンを大きく配置。アップデート初回起動時に変更点ダイアログを表示（PHASE 8 で実装）。
時系列タブの「平均値」が、測定点数の少ない日では信頼性が低い 中 UI 上で count も併記し、点数が少ない日はマーカーを薄く描画。
作業ログのオフラインキューが復帰時に重複登録される 高 クライアント側で local_id (UUID) を発行し、サーバー側で同一 local_id の重複登録を抑止する Idempotency-Key ヘッダ対応を追加（PHASE 6 内）。
Ca/K/Mg モデル未実装のためタブが空転する 低 UI で disable 表示。データ未取得時は「準備中」と明示。
圃場詳細マップタブの統合により、現行の「結果閲覧」フローを使っていたユーザーが戸惑う 中 初回起動時の変更案内ダイアログで「結果閲覧は圃場 > 圃場詳細 > マップ から見られます」を明示。
固定地点機能を後から導入したくなった場合の互換性 低 本書スキーマで analysis_results にカラム追加なしで導入可能。将来 anchor_points テーブルを別途設計する余地を残してある。
